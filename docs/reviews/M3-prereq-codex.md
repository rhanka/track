# ADVERSARIAL REVIEW (Codex 5.5xhigh) — M3-prereq: under-lock clientToken dedup in EventStore.appendCommand

_Date: 2026-06-10 · Verdict: **BLOCK** (4 MUST-FIX). Adjudicated by the conductor — see docs/reviews/M3-prereq-SYNTHESIS.md._

**Findings**

1. MUST-FIX: Cross-workspace suppression is possible. The store assumes aggregateIds “always differ” across workspaces, but that is not enforced. `aggregateId` is just a string/`Ulid` type alias ([types.ts](/home/antoinefa/src/track/src/events/types.ts:5), [types.ts](/home/antoinefa/src/track/src/events/types.ts:115)); IDs come from injectable `newId` ([track.ts](/home/antoinefa/src/track/src/track.ts:77), [track.ts](/home/antoinefa/src/track/src/track.ts:118), [track.ts](/home/antoinefa/src/track/src/track.ts:159)); ingest also exposes `newId` injection ([ingest.ts](/home/antoinefa/src/track/src/ingest/ingest.ts:37), [ingest.ts](/home/antoinefa/src/track/src/ingest/ingest.ts:329)). `validate()` only enforces one aggregate kind per aggregateId, not global uniqueness or workspace uniqueness ([validate.ts](/home/antoinefa/src/track/src/events/validate.ts:104)). So fixed/deterministic IDs, direct store callers, import-like writers, or an actual ULID collision can make V and W share an aggregateId. Then `(clientToken, aggregateId)` in [store.ts](/home/antoinefa/src/track/src/events/store.ts:203) can suppress W because V already wrote that pair. This is not equivalent to the per-workspace `tokenIndex` in [ingest.ts](/home/antoinefa/src/track/src/ingest/ingest.ts:302).

2. MUST-FIX: Same token + different body is silently accepted as a duplicate. `dedupByClientToken()` checks only token equality and aggregateId membership ([store.ts](/home/antoinefa/src/track/src/events/store.ts:188), [store.ts](/home/antoinefa/src/track/src/events/store.ts:203)); it does not compare type/payload/by/prov/id/at/count. The test even blesses this: the second `evt()` has fresh id/time/payload counter, yet is deduped as original ([store.test.ts](/home/antoinefa/src/track/src/events/store.test.ts:24), [store.test.ts](/home/antoinefa/src/track/src/events/store.test.ts:147)). That conflicts with the M3 HTTP design requiring `409` for same idempotency key with different body ([M3-channel-DESIGN.md](/home/antoinefa/src/track/docs/plan/M3-channel-DESIGN.md:49)). If the gateway owns body-digest conflict detection, this store path must be documented as trusting an already-validated digest/namespace; as written it masks divergent writes.

3. MUST-FIX: Dedup bypasses integrity validation. `appendCommand()` returns deduped originals before `validate(existing, readHead(...))` runs ([store.ts](/home/antoinefa/src/track/src/events/store.ts:103), [store.ts](/home/antoinefa/src/track/src/events/store.ts:106)). A log with a content-hash tamper or head truncation/mismatch can return rc=0 on a duplicate token path instead of failing closed. For a valid log, bypassing the append receipt is fine because the event already persisted; for an invalid log, the “original” is not a trustworthy receipt.

4. MUST-FIX: Returned event shape is not faithful once a token appears on more than one command. The implementation returns every persisted event with the same token ([store.ts](/home/antoinefa/src/track/src/events/store.ts:195)), not the matching command, not the matching aggregates, and not a contiguous suffix despite the comment ([store.ts](/home/antoinefa/src/track/src/events/store.ts:175)). The store explicitly allows same token on a different aggregate to append ([store.test.ts](/home/antoinefa/src/track/src/events/store.test.ts:165)); a later retry of the second aggregate will return both the first and second events. That can break receipt count/order expectations and can disclose another workspace’s event if the shared-token events came from different workspaces.

5. SHOULD-FIX: Partial-match behavior is only safe for a true retry of one atomic batch. A physically partial multi-event batch should be rejected by batch validation ([validate.ts](/home/antoinefa/src/track/src/events/validate.ts:187)), but because dedup runs before validation, this safety depends on falling through to validation. A valid log with token reuse over an overlapping aggregate set still appends all inputs, including already-present pairs ([store.ts](/home/antoinefa/src/track/src/events/store.ts:203)). This is the same conflict-detection gap as finding 2.

6. NIT: Frozen event contract shape/hash/seq/head are not changed by this diff. Old logs remain byte-identical unless new tokened events are appended. The contract regression is behavioral: success can now return before validation/head checks.

**Verdict: BLOCK**

Do not ship this as the M3 prerequisite until dedup is workspace-equivalent by construction, conflict-detects or is gated by a durable digest record, validates existing log/head before returning success, and returns exactly the original command’s receipt shape.
