# v2.2a — Design: dependency `resolutionRule` extensions (`linked-accepted`, `decision-settled`)

> Design sub-lot of PLAN-v2 Lot v2.2 — **no code**; specifies the semantics before implementation (v2.2b). Double-consulted **Codex gpt-5.5 xhigh + Opus 4.8** with explicit state-of-the-art grounding; both converged on **A / hybrid-A**. Awaiting owner sign-off on the product policies in §5.

## 1. Problem

A `linked-accepted` dependency blocker (B blocks A until A's ref R is *accepted*) must resolve/re-open as R's acceptance changes. But:
- **acceptance is computed + revocable + commit-relative** — `acceptanceStatus(state, R, baselineCommit)` ∈ `pass|fail|unknown|stale|waived`; a red run regresses a prior pass (SPEC §2.4).
- **`blocker.open` is a fold-time, commit-agnostic boolean** — `isOpen(blocker, items)` in `fold()`, which has no `baselineCommit` (`state/fold.ts`). The MVP throws on `linked-accepted` because it "would never resolve" under the fold model (`track.ts`).

So `linked-accepted` is the one rule whose openness lives at projection time `(state, commit)`, not fold time `(state)`. **That boundary is the design object** — not the auto-resolve mechanics.

## 2. State of the art (why A)

- **CQRS / event-sourcing fact-vs-projection.** Events are facts; derived, revocable values are recomputed at read, never latched into the log (the denormalization trap). `acceptanceStatus` is already derive-at-read; `linked-accepted` openness must join it there.
- **Datomic `as-of` / bitemporal.** Truth is a query over a DB *value* at an explicit basis. track is already bitemporal: **stream order** (transaction time) × **`baselineCommit`** (a valid-time-like "as of which test-world"). Anything depending on `baselineCommit` is a projection, not a fold scalar (XTDB / SQL:2011 / Snodgrass).
- **The decisive analogy: `linked-accepted` ≈ a CI required-check, not a Jira link.** Jira/GitHub/Azure "blocked-by" links gate on *realization done* and do **not** auto-reopen on a downstream test regression (that's `linked-done` / `manual`). But CI/merge gates **do** re-block on a red or stale check against the current commit. `linked-accepted` is semantically a CI gate. No mainstream tracker offers a *revocable* "blocked-until-accepted" relation → this is genuinely novel, and user intuition for a *dependency* must be set explicitly (it re-opens on regression).
- **Latch (B) = enable-wins/ignore-disable flag** — it deliberately discards revocation, which §2.4 says acceptance must keep.

## 3. Decision — **hybrid-A: derive at projection, never fold**

Adopt A's semantics (revocable, commit-relative, auto-re-opens on regression), made type-honest:

1. **`fold` stays pure & baseline-free.** It never calls `acceptanceStatus`. A `linked-accepted` blocker is **not** given a meaningful fold `open` boolean — it is marked as derived (sentinel `open: 'derived'`, or simply dispatched on `resolutionRule`), so no consumer mistakes a fold scalar for its truth.
2. **One projection function** beside `acceptanceStatus`:
   ```
   effectiveBlockerOpen(state, blocker, baselineCommit): boolean
     if blocker.resolvedByEvent: return false              // manual/decision hard-close wins first
     switch (effective rule):
       manual:          return !resolvedByEvent            // (already false above)
       linked-done:     return ref.realization !== 'done'  // commit-agnostic, terminal/stable
       decision-settled:return decision(ref).outcome ∉ {go, no-go}   // commit-agnostic, non-revocable
       linked-accepted: return acceptanceStatus(state, ref, baselineCommit) ∉ ACCEPTED_SET
   ```
3. **Route `openBlockersForItem` / `bucketOf` through it** — `bucketOf` already holds `config.baselineCommit` (`report/buckets.ts`), so AWAITED simply gains the commit it already carried for the `requireAccepted` DONE path. **No new commit parameter crosses any boundary that doesn't already have one.** The v2.0 read contract (`TrackReader`, which already threads `baselineCommit`) is untouched.
4. **"Re-open" = projection re-open, no event.** R `pass` at C ⇒ A not AWAITED; a later `fail` (or asking at a baseline where R is `stale`) ⇒ A AWAITED again; back to `pass` ⇒ closed. Zero new event types, zero latch.

**Why hybrid over pure-A:** pure-A leaves `blocker.open` a boolean fold sets meaninglessly for `linked-accepted`; the `'derived'` sentinel + single dispatch make the bimodality type-enforced, not latent.

## 4. `decision-settled` (straightforward)

- Open iff the referenced **Decision**'s `outcome ∉ {go, no-go}` (`pending`/`deferred` open). Commit-agnostic, non-revocable (terminal outcomes can't transition).
- The existing `kind: "decision"` blocker is the SAME semantic — treat its **effective rule** as `decision-settled`. The enum becomes an *effective* set `linked-done | linked-accepted | manual | decision-settled`; **validate ref shape by rule** (item refs for linked/manual; a Decision ref for `decision-settled`). Decision blockers keep resolving via their go/no-go batch exactly as today.

## 5. Product policies — **owner's call** (defaults below)

| # | Policy | Recommended default | Why / dissent |
|---|---|---|---|
| **P1** | `linked-accepted` = **standing invariant** (re-opens on regression) vs one-time latch | **Standing/revocable** (both experts) | If you ever want a latch too, model it as a *separate* rule (`linked-once-accepted`), never overload "accepted". |
| **P2** | `stale` holds or clears the gate | **Holds (open)** (both) | Strict: AWAITED until proven *at this commit*. "Clears" trades safety for less churn on active branches. |
| **P3** | **`waived` clears or holds** | **✅ DECIDED by owner: HOLDS (strict pass-only)** | `ACCEPTED_SET = {pass}`. Consistent with `requireAccepted` (`=== 'pass'`). A waiver-driven unblock stays an explicit act (manual resolve / future rule), not an implicit alias for pass. |
| **P4** | `linked-accepted` = acceptance `pass` only, or realization `done` **and** accepted | **Pass only** (both) | "done+accepted" = a separate composite rule if needed. |
| **P5** | Ref ends `cancelled`/`rejected` | Inherit `linked-done`'s current behavior (stays open unless ref reaches the bar) — product-visible | Pre-existing open question (`fold.ts` note); not introduced here. |

`ACCEPTED_SET` = `{pass}` under P3-default (strict), or `{pass, waived}` if the owner chooses waived-clears.

## 6. Scope — v2.2b (shipped) vs v2.2c (deferred)

**v2.2b ships ONLY the novel/reviewed core:** the `effectiveBlockerOpen` projection + `bucketOf` reroute; lift the `linked-accepted` throw (facade + **CLI** `--rule` enum); `linked-accepted` reachable end-to-end. Tests: regression re-opens AWAITED at a baseline; pass closes; strict pass-only (waived/stale/unknown/fail HOLD); fold stays baseline-free; settle-once unaffected; CLI + `Track.report` e2e; A1–A7 stay green.

**v2.2c (resolved — reversible calls taken + traced per working-loop hygiene):**
- ✅ **`validate` desync fix-it hint** — BUILT. Each `DesyncFinding` now carries a `hint` (remediation suggestion); `validate` stays detect-only (never applies it). `src/cli/desync.ts`.
- ⏸️ **`decision-settled`** — NOT built. Reversible call: already fully covered by `kind:"decision"` blockers + the go/no-go batch that resolves them; a redundant dependency-rule alias adds surface for no behaviour. Revisit only if a concrete "dependency on a Decision that is NOT the standard decision-blocker" case appears.
- ⏸️ **`report.requireAccepted` per-workspace** — NOT built. Reversible call: the per-invocation `--require-accepted` flag already covers the need; a per-workspace config would introduce a whole config-store concept (events or a `.track/config`) for a speculative requirement. Defer until a real per-workspace policy is requested.
- ⏸️ **BRANCH-import gate sub-checkbox semantics** — NOT changed. Reversible call: SPEC §5 deliberately IGNORES gate sub-checkboxes (`- [ ] make test` under a lot); importing them as criteria/sub-items would contradict the frozen spec and conflate gates with UAT. Keep ignoring; revisit as an explicit SPEC amendment if needed.
