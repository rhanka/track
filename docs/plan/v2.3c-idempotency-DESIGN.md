# Lot v2.3c — ingest idempotency (`clientToken`)

Status: **design, double-consulted → decided.** Codex gpt-5.5 xhigh + Opus 4.8 both recommend **Option A**
(an in-log idempotency token) and reject B/C, defer D. Closes the at-least-once footgun shipped in 0.3.0.

## Decision: A — per-WorkEvent `clientToken`, additive hash-covered `EventCore.clientToken`, ingest skips a seen token
- **Why A (both consults):** it is the only option that makes a retry idempotent for **every** kind with one
  rule — creates (no duplicate ULID aggregates), transition kinds (no spurious `illegal transition` throw on
  replay), `priority.assess`/`criterion`/`link` (no duplicate appends). B (create-only dedup) is a **trap**
  (leaves transitions throwing + criterion/link duplicating); C (external sidecar) recreates the desync class
  the hash-chain/freshness work fights; the ledger *is* the log. D (whole-file atomic batch) is complementary
  but bigger and unnecessary once A lands.
- **Mechanism:** `WorkEvent` gains optional envelope `clientToken?: string`; ingest stamps it onto the
  `EventCore` of **every** event it emits for that WorkEvent (incl. multi-event batches like
  `decision.create`). Before applying, ingest consults a token→result-id index built from the log; if the
  token is already present, it **skips** (applies nothing) and returns the **original** assigned id (stable
  ids across retries — what M3/HTTP needs). Absent token ⇒ today's at-least-once behavior (the human
  `track ingest` path is unchanged).
- **The token index is SCOPED PER WORKSPACE** (keyed on the affected aggregate's workspace, resolved from
  the fold). A token used in workspace V therefore cannot suppress (skip) a write on a channel pinned to W,
  nor disclose V's id to W — closing a cross-tenant write-suppression vector that a shared global namespace
  would open once M3 admits multiple authenticated principals onto one log (review finding, applied).
- **Frozen-contract risk: low, precedented.** `clientToken` is an additive, hash-covered core field exactly
  like `prov` (0.2.0): `canonicalize` drops `undefined`, so existing events hash identically — no new event
  type, no `seq`/`prevHash`/chain change, fold unchanged. Hash-covered because a tamperable token changes
  replay/skip behavior and must be detectable. It is **per-event-varying** (unlike `prov`'s per-channel
  snapshot) — correct for a delivery key, and NOT the per-event-trust smell that got `proposed` removed
  (an idempotency key conveys zero authority: the worst a forged/colliding token does is skip the
  producer's *own* write).

## Scope (this lot)
- `EventCore.clientToken?` (additive, hash-covered).
- `WorkEvent.clientToken?` envelope key (+ `WORK_EVENT_ENVELOPE_KEYS`); validated as a non-empty bounded
  string by `mapWorkEvent`.
- `Track.withClientToken(token, fn)` — a scoped wrapper so `emitBatch` stamps the token on the batch
  without widening ~14 method signatures.
- `ingest`: a token→result-id index from `store.readAll()`; skip-with-stable-id; record applied tokens so
  intra-file duplicates also skip.
- Tests: re-ingest a tokened file ⇒ event count + ids stable; partial-failure-then-retry ⇒ completes, no
  dup; transition replay (tokened) ⇒ no throw; duplicate `assess`/`criterion`/`link` prevented; tampered
  token detected (contentHash); mixed tokened/untokened; old no-token logs still valid.

## Explicitly deferred (noted, not built)
- **Concurrent-retry race (→ M3).** The pre-apply skip is **single-process-correct** (the named M2b
  consumer — a human or sequential CI ingest — is not concurrent). Two *parallel* retries could both see
  "absent" before either commits. Codex's fix — an **in-lock token recheck inside `appendCommand`** plus
  facade stable-id-on-skip — is the right hardening but only matters under parallel automated retries, which
  arrive with **M3** (the authenticated server-to-server channel). M3 maps an HTTP idempotency key to
  per-WorkEvent tokens (e.g. `requestId#index` so a partially-committed request retries cleanly) and adds
  the in-lock recheck. Deferring it keeps this lot minimal without the facade stable-id-on-skip surgery.
- **Token-reuse conflict** (same token, different content) detection; **required** tokens; whole-file
  atomicity (D); `sourceKey` create dedup (B). Standard `Idempotency-Key` semantics: the producer owns
  token uniqueness; M2b skips on token-seen.
